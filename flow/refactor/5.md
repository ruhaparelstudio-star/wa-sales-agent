You are implementing Phase 5 of a chat-agent refactor.

## Objective
Introduce field-level booking candidate validation and confirmation gating before data is persisted.

## Problem
Booking fields are currently captured too heuristically. This can cause incorrect data storage and stage jumps.

## Scope
Implement:
1. BookingFieldCandidate DTO usage
2. BookingFieldValidator service
3. BookingConfirmationPolicy service
4. persistence gate before LeadBookingDataService writes confirmed data

## Required behavior
- extracted booking fields must first become candidates
- each candidate must have:
  - field_name
  - raw_value
  - normalized_value
  - confidence
  - source
  - status
  - requires_confirmation
  - validation object
- low-confidence or ambiguous candidates must not be persisted as confirmed facts
- the system should request clarification or confirmation when needed
- stage progression must respect confirmation status for critical booking fields

## Critical fields to handle first
- event_date
- location
- service_type

## Constraints
- additive refactor only
- do not rewrite full slot extraction model in this phase
- preserve existing extraction where possible
- insert validation layer between extraction and persistence

## Tests required
1. valid event_date candidate passes validation
2. ambiguous date requires confirmation
3. invalid location candidate blocked
4. confirmed candidate can persist
5. unconfirmed critical candidate blocks stage progression
6. clarification payload generated correctly

## Out of scope
- full slot extraction redesign
- full lead memory redesign
- generic response rewrite

## Deliverables
1. validator service
2. confirmation policy
3. integration with booking persistence path
4. tests
5. summary of changed booking flow

## Definition of Done
- booking writes now pass through candidate validation
- critical fields no longer silently persist when uncertain
- stage progression respects confirmation gate
