Executive Summary

Sistem ini adalah multi-tenant wedding sales platform dengan frontliner AI di WhatsApp, namun tetap mempertahankan intervensi manusia pada momen yang berisiko tinggi atau memerlukan otorisasi final. Fokus utamanya bukan sekadar menjawab chat, tetapi mengubah chat masuk menjadi lead yang rapi, terklasifikasi, di-follow-up dengan aman, lalu dihandover ke manusia pada saat yang tepat.

Inbound-first Human handoff wajib GPT-4.1 Mini single-model Multi WA agent via QR (Baileys) Manual payment approval Invoice client + billing invoice terpisah Notification system ✦ Media handling ✦ No campaign mode in MVP
Masalah yang Diselesaikan
  • Chat masuk menumpuk dan lambat dibalas.
  • Lead sering hilang karena follow-up berantakan.
  • Vendor tidak punya lead tracking dan history rapi.
  • Admin capek jawab pertanyaan berulang.
  • Tidak ada alert saat ada lead hot atau billing jatuh tempo.
Nilai Produk
  • Fast response dengan tone natural.
  • Lead qualification dan memory otomatis.
  • Invoice delivery workflow.
  • Subscription & billing manual-friendly.
  • Notifikasi proaktif ke admin via email + in-app.
  • Media (gambar, dokumen) tersimpan rapi.
Boundary Sistem
  • Availability final ditangani manusia.
  • Custom package oleh manusia.
  • Verifikasi DP/pembayaran oleh manusia.
  • No auto blast / no campaign di MVP.
  • Voice note tidak di-transcribe di MVP.
  • Image tidak di-analyze di MVP.

Anti-Ban Guardrails & Safety Logic

Karena pairing WA memakai QR/session via Baileys (non-official), sistem harus mengutamakan perilaku yang aman: inbound-first, tidak blast, follow-up terbatas, stop condition jelas.

Inbound-first
  • Agent hanya auto-reply pada chat yang memang masuk.
  • Tidak kirim pesan pertama ke nomor dingin.
  • No campaign / no blast di MVP.
Follow-up Limit
  • Maksimal 2 auto follow-up per inquiry.
  • FU-1 minimal 18 jam setelah last message.
  • FU-2 minimal 48 jam setelah FU-1.
  • Setelah itu stop total.
Natural Reply Delay
  • 2–5 detik untuk jawaban pendek.
  • 4–10 detik untuk rekomendasi paket.
  • Typing window untuk bundling pesan user.
Adaptive Throughput
  • Per-lead pacing: jangan bombard 1 lead terlalu rapat.
  • Per-number rolling window: batasi pola burst.
  • Priority queue: realtime reply > invoice send > auto follow-up.
  • Adaptive cooldown: follow-up ditunda jika nomor sedang sibuk.
Stop Conditions
  • User minta stop / tidak minat / hapus nomor.
  • Sentiment negatif berat atau komplain.
  • 2 follow-up tidak dibalas.
  • Lead sudah handoff ke admin.
  • Agent disconnected.
  • Conversation risk score melewati threshold.
Outbound pipeline: compose response → media check (jika inbound media, skip LLM) → guardrail policy check → lead state check → quiet hours check → follow-up limit check → risk score check → priority queue (high/medium/low) → adaptive throttle per agent number → delay engine (natural typing simulation) → Baileys send → update message status

QR Pairing & Multi WhatsApp Agent (Baileys)

Pairing Lifecycle
Admin click "Tambah Agent" → Check plan.max_agents vs connected agents (slot check) → Create pairing session (DB) → POST /agents/:id/start ke Baileys → Browser connect SSE /baileys/agents/:id/qr-stream → QR tampil di modal (base64 image) → QR expires → Baileys generate baru → SSE update otomatis → If page closed: POST /agents/:id/cancel → stop QR → If scanned: Baileys POST webhook agent_connected ke Laravel → WhatsApp agent status = connected, session tersimpan di filesystem
Rules yang Terkunci
  • 1 QR = 1 pairing session, tidak reusable.
  • QR auto-generate hanya selama admin di halaman (SSE active).
  • Admin keluar halaman → cancel endpoint dipanggil → stop QR.
  • Nomor connected → scan ulang ditolak.
  • Nomor disconnected → scan ulang = reconnect (pakai existing record).
  • Slot dihitung ulang setelah status berubah ke connected.
  • Session tersimpan di filesystem Baileys (persistent across restart).

LLM Strategy (Single Model)

Sistem memakai GPT-4.1 Mini sebagai satu model untuk semua task, dengan mode prompt berbeda agar tetap efisien. LLM hanya dipanggil untuk text messages — media inbound ditangani di layer sebelumnya.

Mode 1 — Classifier
  • Intent
  • Sentiment / emotion
  • Field extraction
  • needs_handoff
  • Output: JSON ringan
Mode 2 — Sales Brain
  • Jawab user
  • Rekomendasi paket
  • Handle objection ringan
  • Follow-up
  • Output: natural text
Mode 3 — Admin Summary
  • Ringkasan handoff
  • Recap percakapan
  • Context untuk takeover admin
  • memory_updates, follow_up_eligible
Context assembly order (context-assembly.md): 1. Tenant policy snapshot 2. Lead memory snapshot 3. Booking schema missing fields (if needed) 4. Relevant knowledge subset (max 3 items) 5. Recent messages (max 6) 6. Conversation summary 7. Current task mode instruction Never send: full history, full knowledge dump, cross-tenant data

Appendix: Rules yang Sudah Terkunci

Operational Rules
  • Availability final ditangani manusia.
  • Custom package ditangani manusia.
  • Verifikasi DP/pembayaran ditangani manusia.
  • Portfolio diarahkan ke kanal seperti Instagram, lalu handoff jika perlu.
  • Media non-text disimpan; AI hanya proses text.
WhatsApp Agent Rules
  • Connected tidak expired sampai admin disconnect.
  • Disconnected dianggap tidak terdaftar dan tidak mengurangi slot.
  • Scan ulang nomor disconnected = reconnect (pakai record lama).
  • Scan ulang nomor connected = reject.
  • Session Baileys tersimpan di filesystem (persistent across restart).
QR Pairing Rules
  • Auto-generate QR baru hanya jika admin masih di halaman pairing (SSE active).
  • Keluar halaman/close/cancel → POST /agents/:id/cancel → stop QR generation.
  • QR tidak reusable.
  • QR rotate tiap ~20 detik secara otomatis via Baileys.
Subscription Rules
  • Plan di-assign manual oleh super admin.
  • Payment subscription diverifikasi manual.
  • Billing invoice subscription dibuat otomatis H-7 sebelum expired.
  • Dashboard vendor harus menampilkan warning sebelum masa aktif habis.
  • Notifikasi email dikirim di H-7, H-3, H-1, H-0, H+1.
Anti-Ban Rules
  • Inbound-first only.
  • Maksimal 2 follow-up per inquiry.
  • Delay natural dan bundling pesan masuk.
  • Adaptive throttle per nomor agent, bukan fixed global delay.
  • Stop automation saat user negatif, opt-out, atau handoff.
LLM Rules
  • 1 model only: gpt-4.1-mini.
  • Never send full chat history.
  • Context: recent 6 messages + summary + lead memory + max 3 knowledge items.
  • Classifier → Response → Summary (3 mode terpisah).
  • Setiap LLM call dicatat ke llm_usage_logs.
  • Soft cap per plan sebagai warning threshold.
Media Handling Rules (v2)
  • AI hanya proses text. Media disimpan, tidak di-analyze di MVP.
  • Image/document setelah invoice sent → auto-detect sebagai bukti bayar.
  • Voice note tidak di-transcribe di MVP.
  • Sticker dan reaction diabaikan.
Notification Rules (v2)
  • Semua notification via queue (ShouldQueue).
  • database channel: semua notif. mail channel: event kritis saja.
  • Admin tidak perlu buka dashboard untuk tahu event penting.
